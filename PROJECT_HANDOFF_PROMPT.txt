Core product concept:
Plan the season -> record what actually happened -> automatically shift future task timing when actual dates differ from planned dates.

Primary mental model:
- Farm -> Field -> Block -> Bed
- Crop plan records are parent plantings.
- Actual bed usage can be split into placements.
- The map is central. The user wants to build plans visually on the map and record work by clicking map objects.

Long-term product direction:
- The app should represent farm land over time, not only current state.
- Field/block/bed/crop/cover-crop states should be assessable by date using a map time slider.
- Future versions should support area events and point events over time, possibly with high-detail plant placement.
- It should be able to hold detailed plant-level data only when needed, without bloating the data model for normal bed/block-level planning.
- It may eventually export/render data into game engines or 3D environments for robotics.
- It may eventually use photos/drone imagery to refine bed/plant placement.
- It may eventually include seed/genetics catalogs from seed companies, starting with Johnny's and High Mowing, but that is not a current priority.

Major design decisions already made:
- Use an event/timeline-capable model for field state instead of overwriting current state only.
- Preserve planned dates and actual dates.
- Use milestone/flow-chart-based task scheduling instead of fixed-only task dates.
- Keep beds mostly map geometry, not individual plant coordinates.
- Represent exact plant counts as lengths/sections of beds rather than visible individual plant dots by default.
- Show more geometry detail only at closer zooms.
- Roads/paths can be represented in the bed layer but should not be plantable.
- Workers can record task completion; planners/owners can adjust plans.
- Most users should not need to understand the database model; workflows should be visual and obvious.

Accounts and permissions:
- There are farm accounts with username/password.
- There are at least two practical privilege levels:
  - Planner/owner: can plan and adjust farm data.
  - Worker/employee: can record task completion.
- Admin control panel exists or is being built for global/admin functions.
- Feedback/suggestion/problem reporting should be available to all users, but reports should only be visible in admin.
- Maps are visible to all users unless private mode is turned on in settings.

Map system:
- Uses Leaflet with aerial imagery.
- Fields and blocks are polygons.
- Beds are generated within blocks and have geometry stored in PostGIS.
- Blocks and beds can be selected on the map.
- The user wants the map to be mouse-first and practical, not fancy.
- The map has planning and working modes. Planning mode is for layout/crop plan changes. Working mode is for recording actual work from the map.
- The map time control should default to the current week, with week back/forward buttons and a time slider still available.
- Crop labels should only show crops when they are actually in the ground according to the selected map time.
- Crop labels should be short: crop + variety, not status clutter.
- Old/past-year data should not clutter current allocation/task workflows unless the user intentionally looks back in time.

Current map/bed-generator issues needing attention:
- The bed generator still sometimes cuts generated beds short when zoomed into part of a field. The intended behavior is to generate full-length beds based on the selected block geometry, not the current viewport.
- The first-bed offset has been unstable. The backend uses one-sided PostGIS ST_Buffer with side=left/right, so the picked line should generally be treated as a bed edge, not a centerline.
- If beds already exist in a block, adding a new bed should normally build off the last selected/existing bed, not start again from the block edge, unless the user selects “replace existing beds.”
- The direction arrow is now confusing and probably should be removed or simplified, because the tool should know which direction is inward/next.
- Deleting beds from the selection card currently appears not to work and needs debugging.
- Road/path presets were added using bed_presets.is_road and beds.source = 'road'. Roads should show on the map but not be available for planting, placement, or harvest logging.
- The bed generator UI should prefer a simple path:
  - Select block or bed.
  - Choose preset.
  - Make one bed or fill block.
  - Advanced line/curved controls are hidden lower down.
- User wants harvest/access roads. Their farm often skips two bed slots for roads, but other farms may use other systems. A road preset all path/no bed is acceptable and should not be plantable.
- Beds need to store exact geometry in a way that can later be updated from drone/photo/survey data.

Current desired bed workflow:
- If no beds exist or user chooses replace all, build from a selected block edge inward.
- If a bed is selected, “Make one bed” should build the next bed off that selected bed by default.
- After making one bed, select the new bed automatically so repeated clicks keep extending the bed set.
- If filling a block, generate as many beds as fit from the chosen starting edge/bed, respecting preset bed width and path spacing.
- Generated beds should square/clip cleanly at irregular block edges; avoid triangle spikes sticking out.
- If beds planned vs actual differ, the app should eventually support recording actual bed count and squish/stretch path spacing to fit reality.
- Making beds in planning should create a bed-making task for that block, even if no due date is known yet. Once crop dates in the block exist, schedule the task before beds are needed.

Crop planning:
- Annual Crop Plan should be grouped by year and hide older years by default.
- “Parent Planting” was developer terminology and is confusing to the user. In UI, avoid “parent/child placement” language where possible.
- Current user-facing planting form should read “Plan Planting.”
- Use “Planting date,” not “watered in date.”
- Crop and variety dropdowns should be prominent.
- Lot number should be a free text field around 128 characters.
- Seed/genetics structure desired:
  - Higher categories can be hidden or auto-filled.
  - User-facing fields should include crop, variety, breed, supplier, lot number, possibly catalog number.
  - Levels should be skippable.
  - Example hierarchy: Nightshade -> pepper -> jalapeno -> variety/breed -> supplier -> lot number.
- For transplants:
  - Include tray count calculator.
  - Standard tray sizes: 128, 288, 72, 50.
  - Tray location should be optional and eventually map-based.
- For direct seeding:
  - Use seed spacing x bed length.
- Intended bed/location should use dropdowns: field/block first, then bed.
- Spacing should be two numeric boxes in inch or cm depending on metric/ft setting.

Bed allocation from map:
- User wants to select a bed on the map, choose a likely planting from a dropdown, and allocate plants visually.
- Dropdown should prioritize:
  - right field + right date first
  - right date + nearby fields next
- Slider should adjust plant count and bed length/sections.
- Slider should start around 90% of available seeded/transplant count.
- It should subtract already placed plants, but allow fudge room up to roughly 20% because actual germination/transplant counts vary.
- There should be a clear way to say whether the allocation uses all remaining plants or only part.
- Placement should fill beds from one side and wrap to the next bed with no space unless specified.
- At close zoom, show sections of bed by crop. At farther zoom, hide individual beds and show summarized crop/block labels.

Tasks and task flows:
- Task flow chart system exists and was reworked to be more visual.
- The user wants task flows as editable visual flow charts, not tangled lists.
- Nodes can be connected up/downstream.
- Node details should be editable on the node itself via an edit icon/expanded fields.
- Task flow nodes should support icons and two editable colors.
- An 8-bit tractor PNG exists in demo data and has been used or should be used for task icons. The 3/4 tractor view should be primary in forms/task lists. On the map, tractor icons should move along beds needing work that week, using closest directional view and flipping sprites if useful.
- Task icons should appear in task flow, task list, and map.
- Current tasks should focus on current year/week by default. Past years should not bog down current task list.
- Task list has week back/forward buttons and should default to current week.
- Marking tasks done should be easy.
- If a task is scheduled within +/-4 weeks, record date defaults to today. If older, it defaults to planned week/date.
- Marking planting in the future is not allowed.
- Recording from a task should preserve enough information to keep the system useful, especially for planting/transplanting tasks.
- There was a bug where task recalculation deleted/recreated tasks and broke identity/history. This was addressed by updating tasks by flow-node identity rather than destroying them. Preserve task identity going forward.

Record work:
- Workers should be able to record completion.
- Planner can adjust planting details while recording from a task.
- For “plant in field” task recording:
  - Date is near top and defaults appropriately.
  - Notes should be at bottom.
  - Remove confusing “adjust planting from...” wording.
  - Bed placement should autofill from plan but be editable via field/block/bed dropdowns.
  - Spacing should be two numeric boxes in inch/cm.
  - Plant count should calculate bed length based on in-row spacing.
  - Recording should withdraw from available transplants with some fudge room.

Cover crop and field/block state:
- Cover crop is important and should be treated as land/state planning, not normal harvested crop.
- Blocks are not always fully prepped; blocks may be split into beds, cover crop, unassigned/empty parts.
- The user disliked drawing polygons for every cover crop case.
- Desired cover-crop/block-state workflows:
  - Fill whole block with a state/use.
  - Or split a block by a two-point line whose endpoints are on the block edge.
  - The unused split-off part should be “unassigned,” not “reserve.”
- Blocks should have current state assigned when made, and may separately have future use assignments.
- Current state needs time/date history.
- Cover crop needs planned dates and actual work dates.
- Readiness/cleanup states should include needs mowing, plastic beds need removing, and notes for specific cleanup like tomato stakes.
