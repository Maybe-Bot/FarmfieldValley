LOAM LEDGER CLEAN REBUILD PROMPT

Use this prompt while working inside the existing FarmMan / Loam Ledger repository.


MAIN GOAL

Rebuild the complete Loam Ledger site so the finished application looks and works the same as the current active site, on desktop and mobile, but has a clean maintainable structure.

The rebuild must:

1. Preserve every feature and workflow that a user can currently reach and use.
2. Preserve the current visual design, wording, responsive behavior, permissions, and farm workflow.
3. Remove code and features that were built but are not actually used.
4. Break the frontend and backend into sensible feature modules instead of recreating giant files.
5. Preserve existing farm data and database compatibility.
6. Be understandable and maintainable by a future coding agent.

Do not redesign the application. This is a careful clean rebuild, not a new product.


SOURCE OF TRUTH

Use these sources in this order:

1. The current running application and reachable user workflows.
2. The current source code and API behavior.
3. docs/cards-desktop.txt and docs/cards-mobile.txt for the visible card inventory.
4. The current tests, database migrations, README, and beta checklists.

If the documentation disagrees with the running application, follow the running application and update the documentation.

Before changing code, run and record:

- npm test
- npm run build
- git diff --check

Do not overwrite or discard existing uncommitted user changes.


IMPORTANT SAFETY RULES

- Never delete, reset, truncate, or reseed an existing database.
- Never change the meaning of existing farm records during this rebuild.
- Keep existing migrations intact. Add a migration only when a real schema change is required.
- Spreadsheet imports must remain append-only. They must never delete existing farm records and must not offer a delete/replace option.
- Keep full backup export/import compatibility for existing farm data.
- Preserve authentication, sessions, CSRF protection, CORS handling, farm scoping, planner/worker permissions, admin permissions, undo/redo, feedback reports, inbox replies, and usage tracking.
- Do not remove a route, table, type, or helper merely because its name looks old. Prove it is unreachable and not required for existing saved data, imports, migrations, tests, or compatibility.
- Do not put the whole rebuild into one new App.tsx or one new server.ts.
- Do not change user-facing labels or move controls unless required to reproduce the current mobile or desktop layout.

Frontend architecture rules:

- App.tsx should compose providers, navigation, and page components. It must not contain thousands of lines of feature implementation.
- Each page belongs to its feature folder.
- Each major visible card should be its own focused component.
- Put reusable stateful behavior in hooks only when it is genuinely shared.
- Keep map state together in a map controller/reducer instead of dozens of unrelated top-level useState calls.
- Separate map geometry calculations from React rendering.
- Separate API calls from page components.
- Centralize shared domain types and permission checks.
- Use one source of truth for labels, task types, vehicle types, themes, and navigation.
- Split CSS by design tokens, shared layout, responsive rules, and feature styles. Do not replace the current visual design with a component library.
- Avoid generic abstractions that hide farm-specific behavior.
- Keep components small enough that their purpose is clear. As a guideline, investigate any component above roughly 400 lines and split it when it has more than one responsibility.

Backend architecture rules:

- Route files should validate input, enforce permission, call a service, and format the response.
- Business rules belong in services/domain modules, not giant route handlers.
- Database queries belong near their feature and should remain farm-scoped.
- Keep task scheduling and task-flow validation independent and testable.
- Keep map geometry and bed generation independent and testable.
- Keep spreadsheet parsing, validation, append-only import, and export separate.
- Keep authentication and account-email behavior separate from farm workflows.
- Use consistent error handling and validation.
- Do not rewrite working SQL merely to make it look different.

DEAD-CODE AUDIT METHOD

Before deleting anything:

1. Inventory all pages reachable through navigation, buttons, table rows, map selections, tutorials, permission-specific controls, and modal actions.
2. Test planner, worker, signed-out, and admin paths.
3. Search every frontend API method for callers.
4. Search every backend route for frontend, script, test, import/export, or deployment callers.
5. Search every exported function/type/component for runtime and test imports.
6. Search CSS selectors against rendered class names.
7. Check package scripts and deployment documentation.
8. Check database migrations and backup import/export compatibility.
9. Only remove an item after this evidence shows that it is not active.

If uncertain, keep compatibility behavior isolated behind a clearly named adapter and document why.


IMPLEMENTATION ORDER

Complete this as a controlled replacement, not a one-step rewrite.

Phase 1: Baseline

- Run tests and build.
- Record active navigation and permission-specific workflows.
- Capture desktop and mobile screenshots of every reachable screen and major conditional card.
- Create a feature/reachability inventory.
- Create a list of removal candidates with evidence.

Phase 2: Shared foundation

- Extract API client modules.
- Extract shared types, formatting, permissions, buttons, cards, tables, form controls, modals, and responsive layout primitives.
- Split design tokens and global styles without changing appearance.

Phase 3: Feature extraction

- Rebuild one feature at a time.
- Keep old and new behavior comparable during each extraction.
- Start with simpler screens such as Inbox, Settings, and Admin.
- Continue with Seed Bank, Crop Plan, Planting Detail, and Tasks.
- Rebuild Task Flows and Garage.
- Rebuild the Farm Map last because it has the most cross-feature state and geometry behavior.

Phase 4: Backend modularization

- Move routes and business rules feature by feature.
- Keep route URLs and response shapes stable unless a coordinated frontend change is tested.
- Preserve database and backup compatibility.

Phase 5: Remove proven dead code

- Remove only items verified by the dead-code audit.
- Include the unused-feature removals listed in this prompt when confirmed.
- Remove obsolete styles and types after their code is gone.

Phase 6: Final verification

- Run automated tests, production builds, and diff checks.
- Complete the current beta workflow checklist.
- Compare current and rebuilt desktop/mobile screenshots.
- Test against a copy of a migrated existing database, not only an empty seeded database.
