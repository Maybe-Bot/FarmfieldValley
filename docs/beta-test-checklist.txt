Loam Ledger Beta Test Checklist

Use this as a practical testing pass before inviting beta users.

Mark each item:
[y] Works
[y1], [y2], [y3], etc. Worked before; code has changed that many rounds since it was last directly retested
[y8] or higher means stale: it worked before, but has gone 8+ edit rounds without being rechecked
[n] Broken
[?] Confusing
[ ] Not tested

When updating this checklist after code changes:
- If the item was directly affected by a change, reset it to [ ] so it gets retested.
- If the item was not changed, increase [y] to [y1], [y1] to [y2], [y2] to [y3], and so on.

Public beta hosting settings:

This is the item that sounded important but unclear.

Plain meaning: before strangers or beta users use the hosted app, the website and API must know where each other live, and the API must use real security settings. If these are wrong, people may not be able to log in, browsers may block the app, or login cookies may be unsafe.

Ask for help checking these before public beta:

[ ] Frontend host has VITE_API_BASE_URL set to the real API address.
[ ] API host has CORS_ALLOWED_ORIGINS set to the real website address.
[ ] API host has PUBLIC_WEB_URL set to the real website address.
[ ] API host has PUBLIC_API_URL set to the real API address.
[ ] API host has NODE_ENV=production.
[ ] API host has a real long CSRF_SECRET, not the example value.
[ ] API host has SESSION_COOKIE_SECURE=true.
[ ] API host has SESSION_COOKIE_SAMESITE=Lax, unless the website/API setup requires None.
[ ] If using SESSION_COOKIE_SAMESITE=None, confirm SESSION_COOKIE_SECURE=true.
[ ] Confirm the production database has automatic backups.
[ ] Confirm you know how to restore a database backup if something goes wrong.
[ ] Confirm nobody runs npm run db:seed on real beta data.
[ ] Confirm migrations run with npm run db:migrate when updating the server.

Final beta decision:
